# Roost — панель управления игровыми серверами

Аналог Pterodactyl: одна панель-мастер + произвольное число нод с демоном
(`wingsd`), общающихся по защищённому протоколу. Полное описание архитектуры,
схемы БД и протокола — в [`docs/`](docs/).

## Документация

| Документ | Содержание |

| [docs/ARCHITECTURE.md](docs/ARCHITECTURE.md) | Структура репозитория, роли компонентов, масштабируемость, безопасность |

## Структура репозитория

```
backend/    панель-мастер (Go) — REST API, JWT/RBAC, WS-шлюз для браузера
daemon/     wingsd — демон на каждой ноде (Go) — Docker, cgroups, консоль
frontend/   SPA (React + Vite + TS + Tailwind), стили — panel.css
docs/       архитектура, схема БД, протокол
```

## Автоматическая установка (Ubuntu 22.04/24.04)

```bash
sudo ./install.sh
```

Первым делом установщик спрашивает язык (English / Русский) — в русском
режиме каждый шаг сопровождается пояснением, зачем он нужен. Дальше —
интерактивное меню (`whiptail`, с текстовым fallback):

1. Установить Мастер-панель (Backend + Frontend + БД)
2. Установить Демон-ноду (Wings + Docker)
3. Установить всё вместе (на одном сервере)
4. Удалить (база данных и файлы серверов сохраняются)
5. **Полное удаление** — сносит базу, Docker и вообще все данные; требует
   ввести `DELETE` заглавными буквами для подтверждения

Ставит Go/Node.js (собственные версии, не устаревшие пакеты Ubuntu), Docker,
PostgreSQL + Redis, собирает backend/frontend/daemon из этого репозитория,
генерирует секреты (JWT/DB пароль/ключ шифрования) в `/etc/panel/panel.env` и
`/etc/wingsd/wingsd.env`, поднимает systemd-юниты (`panel.service`,
`wingsd.service`) и настраивает UFW. Модули лежат в [`scripts/`](scripts/) —
один файл на этап, с функцией `install_<этап>`, подключаемой в `install.sh`.

> Схема БД проекта — PostgreSQL (см. `backend/migrations/0001_init.sql`), а не
> MariaDB — установщик ставит именно её, а не то, что упоминалось в
> черновике промпта для установщика.

При установке панели (шаги 1 и 3) установщик сам, по порядку:

1. **Спрашивает домен** — поддомен и основной домен (например `panel` +
   `example.com`) — прописывает его в nginx и выпускает бесплатный
   HTTPS-сертификат через certbot (Let's Encrypt). Если домен не ввести —
   остаётся на HTTP по IP, сертификат можно добавить позже повторным
   запуском `install.sh`.
2. **Создаёт главного админа** — спрашивает email/логин/пароль (пароль
   вводится скрыто, с подтверждением) и сразу заводит аккаунт через
   `panel-admin` (`backend/cmd/panel-admin` — единственный способ завести
   первого пользователя, раз регистрации через UI нет и не должно быть).
3. В конце печатает адрес панели (`https://ваш-домен` или `http://IP`).

Только после этого, при установке демона (шаги 2 или 3), скрипт объясняет:
зайдите в панель по адресу из шага выше, откройте страницу **Nodes**, нажмите
**Add node** — панель покажет `daemon_token` один раз, его и нужно вставить
в запрошенное скриптом поле. То есть ключ ноды спрашивается только тогда,
когда сайт уже реально работает и в нём можно залогиниться.

## Быстрый старт (локальная разработка)

### 1. База данных
```bash
createdb panel
psql panel -f backend/migrations/0001_init.sql
```

### 2. Панель (backend)
```bash
cd backend
go mod tidy        # уже выполнено, зависимости зафиксированы в go.sum
go run ./cmd/panel # слушает :8080 (см. internal/config для ENV-переменных)
```
Собирается и проходит `go vet` "из коробки". `NodeClient` в
`cmd/panel/main.go` — единственное осознанно недоделанное место: там нужно
подключить резолвинг нод (fqdn/scheme/port + токен) из вашего секрет-хранилища,
см. комментарий в файле.

### 3. Демон (daemon), на каждой ноде
```bash
cd daemon
go mod tidy
export WINGSD_DAEMON_TOKEN=<токен, выданный панелью при создании ноды>
go run ./cmd/wingsd  # слушает :8443, требует доступный Docker daemon
```

### 4. Фронтенд
```bash
cd frontend
npm install
npm run dev   # http://localhost:5173, проксирует /api и /ws на :8080
```
`npm run build` (tsc --strict + vite build) проходит чисто.

## Что уже реализовано в стартовом коде

- **backend**: конфиг, пул PostgreSQL (pgx), JWT-выдача/валидация, bcrypt для
  паролей и токенов нод, RBAC middleware (интерфейс `PermissionChecker` —
  подключите свою реализацию поверх таблиц `role_permissions`/`server_subusers`),
  REST-хендлеры auth/nodes/servers, WS-шлюз для консоли/метрик.
- **daemon**: обёртка над Docker SDK (создание контейнера с лимитами cgroups
  из memory/swap/cpu/io, старт/стоп/kill/remove, стрим логов, attach к stdin,
  снятие stats), HTTP API с bearer-авторизацией по токену ноды, WS-хаб для
  консоли.
- **frontend**: типы, зеркалящие Go-модели, REST-клиент, `ServerList` —
  компонент со списком серверов (карточки, живые метрики по WS, кнопки
  питания), собранный целиком на классах из предоставленного `panel.css`.

## Чего нет и что дальше

Это стартовый каркас, а не готовый продукт. Явно не реализовано (см. TODO/
комментарии в коде): миграция control-plane на gRPC (proto уже готов),
файловый менеджер и бэкапы (эндпоинты описаны в протоколе, обработчики — нет),
установка серверов через install-script, страницы SPA кроме дашборда
(консоль, файлы, БД, расписания, настройки — верстка для них уже есть в
`panel.css`), очередь задач для scheduler'а, интеграция SFTP.
