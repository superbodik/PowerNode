# Arbeit — что ещё можно поправить и добавить

Живой список находок: реальные баги/недоделки уже в коде, плюс идеи для
новых фич. Составлено по результатам сквозного аудита схемы БД и кода —
искал места, где что-то читается/пишется наполовину, где в схеме есть
колонка/таблица без единого использования в Go-коде, и где документация
разошлась с тем, что реально запущено.

Не дублирую то, что уже пофикшено в этой сессии (SFTP, бэкапы, домены,
suspend, capacity ноды, egg-валидация, создание/сброс пароля, TOCTOU-гонки
и т.д. — всё это уже в `add.md` и запушено).

## 1. Реальные баги / недоделки (стоит проверить и поправить)

### 1.1. Activity log жёстко обрезан на 100 записей, без пагинации
`ActivityHandler.List` делает `... LIMIT 100` и всё — ни `OFFSET`, ни
фильтра по дате, ни поиска по пользователю/событию. На активной панели
через какое-то время старые события становятся физически недоступны
через UI (они в БД лежат, но их никогда не покажут). Стоит добавить
пагинацию (`?before=<id>` или `?page=`) и, желательно, фильтр по
серверу/пользователю/типу события — благо `activity_logs` уже
проиндексирован под это (`created_at DESC`, `(server_id, created_at DESC)`).

### 1.2. `servers.status` в БД не обновляется вообще после создания
Уже находили и чинили симптом (застревание online/offline после обрыва
WS), но корень шире: колонка `status` в БД пишется РОВНО один раз — при
создании сервера (`'installing'` → `'offline'`). Все дальнейшие
"running"/"starting"/"stopping", которые видно в интерфейсе — чисто
фронтовая иллюзия поверх WS-статистики, которая живёт только в памяти
браузера и обнуляется при перезагрузке страницы (до следующего сообщения
по WS). Если сервер выключится, пока никто не смотрит на страницу — в
БД он навсегда останется "offline" с последнего раза, даже если через
минуту сам поднимется. Список серверов (`ServerList`) должен уметь это
показать корректно даже без открытой WS-сессии. Разумный фикс: демон
периодически (или сразу при поллинге статы) шлёт панели актуальный
`state`, а панель пишет его в `servers.status` — то есть `Hub.pollStats`
должен не только транслировать в браузер, но и апдейтить БД.

### 1.3. `nodes.agent_version` — колонка есть, никто её не пишет
Ни демон при старте/пинге не сообщает свою версию, ни панель её нигде
не читает. Было бы полезно (и совсем недорого сделать, раз версия
демона теперь честный semver — `daemon/cmd/wingsd` можно вкомпилировать
version так же, как `backend/cmd/panel` уже делает) — тогда на странице
Nodes видно, какие ноды отстали и надо обновить, вместо того чтобы
руками помнить.

### 1.4. CORS `AllowedOrigins` захардкожен на `http://localhost:5173`
`backend/internal/api/router.go`, `cors.Handler`. В проде панель и API
живут на одном домене через nginx, так что для обычной работы браузера
это не мешает (same-origin, CORS вообще не участвует). Но если
когда-нибудь понадобится браузерный доступ к API с другого домена
(например, отдельный дашборд/интеграция стороннего сайта через API-key
прямо из JS в браузере пользователя) — он молча упрётся в CORS. Стоит
либо вынести allowed-origin в конфиг (`PANEL_ALLOWED_ORIGINS`), либо
явно задокументировать, что API рассчитан только на server-to-server
использование ключей, а не на browser-to-API с чужого домена.

### 1.5. Список серверов (`ServerHandler.List`) без пагинации
Не критично при десятках серверов, но при сотнях/тысячах — это один
`SELECT *` без `LIMIT`. Не горит, но если панель когда-то вырастет,
стоит добавить `?limit=&offset=` заранее, а не когда страница начнёт
тормозить.

### 1.6. `docs/PROTOCOL.md` описывает gRPC, которого нет и не было
Документ явно расписывает control-plane на gRPC + mTLS, ссылается на
`daemon/proto/daemon.proto` (файл реально существует, 173 строки) — но
`google.golang.org/grpc` не подключен НИ В ОДНОМ модуле, никакого
сгенерированного `.pb.go` нет, вся реальная связь panel↔wingsd — это
обычный HTTP/JSON + WS с Bearer-токеном (то, что построено и работает
всю эту сессию). Похоже, gRPC был исходным планом, от которого тихо
отказались в пользу более простого HTTP-подхода, а документацию не
поправили. Стоит либо переписать `PROTOCOL.md` под факт (HTTP/JSON + WS,
как оно есть — эндпоинты уже все перечислены в `router.go` с обеих
сторон), либо, если gRPC зачем-то всё ещё в планах — явно пометить
раздел как "не реализовано, roadmap".

### 1.7. Разрешено ли двум людям одновременно писать в один файл?
`FileHandler.Write` просто перезаписывает файл без какой-либо проверки
конфликтов (нет `If-Match`/mtime-check). Два открытых редактора одного
файла — тихо потеряют правки друг друга (last write wins). Не
блокирующий баг, но стоит хотя бя предупреждать в UI, если файл
изменился на диске с момента открытия.

## 2. Что стоит добавить (новые возможности)

### 2.1. Экспорт/скачивание бэкапа панели целиком (не серверов, а самой БД)
Есть бэкапы СЕРВЕРОВ (файлы игровых серверов), но нет ничего для бэкапа
самой панели — `pg_dump` базы + `panel.env`/`wingsd.env` секретов. Если
диск с панелью умрёт, всё (пользователи, ноды, права, история) исчезает
без единого штатного способа восстановления. Даже простой
`scripts/panel-backup.sh` (cron + `pg_dump` в `/var/backups/panel`) закрыл
бы огромный риск для self-hosted инструмента, где нет какого-то
managed-провайдера, который сам всё бэкапит.

### 2.2. Тегерам-уведомления (опционально, если будет SMTP)
Сейчас панель ничего никому не пишет: ни "твой сервер упал", ни "бэкап
не удался", ни "кто-то вошёл в твой аккаунт с нового IP". Для
self-hosted тула это осознанный трейд-офф (не поднимать почтовую
инфраструктуру), но если когда-то захочется — уже есть вся необходимая
активность в `activity_logs`, можно подключить простой SMTP-воркер,
который шлёт дайджест или алерты по важным событиям
(`server.power.kill` из-за OOM, `backup.*` неудача, `node.*` недоступна).

### 2.3. Массовые операции над серверами
Сейчас всё — по одному серверу за раз (start/stop/restart/backup).
Для админа с сотней серверов было бы полезно: выбрать несколько на
Dashboard и разом рестартовать/забекапить/приостановить. Не критично,
но ощутимо ускоряет жизнь на масштабе.

### 2.4. Собственный сертификат для конкретного узла заранее (staging)
Сейчас домен для сервера настраивается ПОСЛЕ создания сервера
(вкладка Domains), и certbot идёт синхронно в момент добавления —
если DNS ещё не разрулился, попытка проваливается и надо пересоздавать
домен заново. Можно сделать "отложенный" статус (`tls_status: pending`)
и фоновый ретрай certbot каждые N минут в течение часа-двух, вместо
одной попытки и всё.

### 2.5. Двухфакторка не обязательна даже для админов
Ни на уровне панели, ни на уровне отдельного пользователя нет опции
"обязательная 2FA для админов". Для панели, которая теперь управляет
SSH-ключами, доменами и бэкапами — это не помешало бы как минимум
опциональной политикой ("is_admin ⇒ обязать включить 2FA при следующем
входе").

### 2.6. Поиск и фильтры на страницах Users/Activity/Nodes
Простой `<input>` фильтр по имени/email/событию — сейчас на этих
страницах ничего, кроме сырого списка (в отличие от Dashboard, где
поиск серверов уже есть).

## 4. Детальный аудит на баги — ПОФИКШЕНО (кроме 4.10)

Прошёлся ещё раз по всем хендлерам `backend/internal/api/handlers/*.go`,
роутам в `router.go` и части демона (`files`, `proxy`, `docker`). Все
находки ниже, кроме явно помеченной 4.10, за один заход почищены,
собраны (`go build`/`go vet`/`gofmt` на обоих модулях + `npm run
build` на фронте) и прогнаны через существующие тесты + разовый
router-smoke-test. Новая миграция `0008_user_token_version.sql`
подхватится автоматически через `update.sh` (таблица
`schema_migrations` уже так устроена).

### 4.1. ✅ `GET /database-hosts` — доступен ЛЮБОМУ юзеру, не только админу
`router.go:116` — `r.Get("/database-hosts", databaseHostHandler.List)` без
`auth.RequireAdmin` (в отличие от Create/Delete на строках 117-118,
которые правильно защищены). А `DatabaseHostHandler.List` вообще не
смотрит на `claims` — просто отдаёт ВСЕ строки `database_hosts`: host,
port, admin_username каждого зарегистрированного MySQL-сервера. Любой
обычный пользователь (у кого просто есть один game-сервер) может дёрнуть
этот эндпоинт и увидеть инфраструктуру всех MySQL-хостов панели (сам
пароль зашифрован и не отдаётся, но host/port/admin_username — да).
Похоже на забытый `RequireAdmin` при рефакторинге — Create/Delete его
получили, List — нет.

**Фикс:** не стал делать `RequireAdmin` — оказалось, обычные юзеры
легитимно используют этот список в `DatabaseManager.tsx` (выбор хоста
при создании своей БД). Вместо этого `DatabaseHostHandler.List` теперь
для не-админов обнуляет `host`/`port`/`admin_username` в ответе,
оставляя только `id`/`name` — ровно то, что нужно для выпадающего
списка, без утечки инфраструктуры.

### 4.2. ✅ `GET /allocations` — та же дыра, плюс пробивает приватность нод
`router.go:156` — `r.Get("/allocations", allocationHandler.List)` тоже
без `RequireAdmin` (Create/Delete на 157-158 защищены). А сам хендлер
(`allocation_handler.go`) не проверяет владение/видимость ноды вообще —
только требует `node_id` в query. Значит любой залогиненный
пользователь может перечислить IP:port конкретной ноды по id, включая
приватные ноды (`is_public = false`), которые он даже не видит в
`GET /nodes` (там фильтрация по `is_public` для не-админов уже сделана
в этой сессии). То есть фича приватности нод, которую сами же
построили, обходится напрямую через allocations.

**Фикс:** та же логика, что и для 4.1 — `CreateServerForm.tsx` легитимно
дёргает `/allocations?node_id=...&free=true`, значит `RequireAdmin`
сломал бы создание сервера. Вместо этого `AllocationHandler.List`
теперь для не-админов проверяет `is_public` запрошенной ноды и отдаёт
404, если нода приватная и не твоя.

### 4.3. ✅ `TwoFAHandler.Setup` тихо выключает уже включённую 2FA — без пароля
`twofa_handler.go: Setup()` доступен с одним валидным JWT/сессией,
никакого запроса текущего пароля. Если у юзера 2FA уже включена
(`totp_enabled = true`) и он (или тот, кто угнал его сессию) дёрнет
`/account/2fa/setup` ещё раз — хендлер молча перезаписывает
`totp_secret` и ставит `totp_enabled = false` (до повторного `/verify`
с новым кодом). Получается, что `Disable` осознанно требует пароль для
выключения 2FA, а `Setup` даёт ровно тот же эффект (2FA выключена) без
пароля вообще. Угнанная сессия (XSS, забытый браузер) может тихо снять
вторую защиту с аккаунта, просто вызвав "пересоздать QR-код".

**Фикс:** `Setup` теперь требует `password` в теле запроса, ЕСЛИ
`totp_enabled` уже true (при первом включении пароль по-прежнему не
нужен — сессия и так уже аутентифицирована, нечего пере-проверять).
Фронт не трогал — кнопка "Enable 2FA" и так рендерится только когда
`!enabled`, так что легитимный путь не задет.

### 4.4. ✅ Создание домена/базы: внешнее действие ДО записи в БД панели — риск "осиротевшего" ресурса
- `ServerDomainHandler.Create` (`server_domain_handler.go`): сначала
  дёргает демона (`client.AddDomain` — реально настраивает nginx +
  certbot на ноде), и только ПОСЛЕ успеха пишет `INSERT INTO
  server_domains`. Если этот INSERT упадёт (конфликт уникальности домена,
  обрыв к БД и т.п.) — домен на ноде уже настроен и получил сертификат,
  но в панели о нём никакой записи нет, значит через UI его никогда не
  найти и не удалить (нет `DELETE`-пути без записи в БД).
- Та же схема в `ServerDatabaseHandler.Create` (`server_database_handler.go`):
  `mysqlhost.Provision` реально создаёт базу+юзера на MySQL-хосте
  ДО `INSERT INTO server_databases`. Если инсерт после этого упадёт —
  на реальном MySQL-сервере остаётся живая база с логином/паролем,
  про которую панель никогда не узнает и не сможет удалить.

**Фикс:** в обоих хендлерах порядок развёрнут — сперва резервируем
строку в БД панели (для доменов: `INSERT ... tls_status='pending'`;
для баз: `INSERT` с уже сгенерированным и зашифрованным паролем), и
только потом дёргаем демона/`mysqlhost.Provision`. Если внешний вызов
падает — компенсирующий `DELETE` по только что созданному id, никакого
осиротевшего домена/базы. Если внешний вызов успешен — `UPDATE`
финального статуса (для доменов) или просто отдаём ответ (для баз,
там нечего обновлять после успеха).

### 4.5. ✅ `UserHandler.Update` защищает только от самопонижения, не от "остаться без админов"
`user_handler.go: Update()` блокирует только `id == claims.UserID`
одновременно со снятием admin/active. Если админов двое, админ A может
снять админку/активность с админа B без всякого предупреждения — если
B был единственным ДРУГИМ админом, в системе остаётся ровно один админ
(сам A), и если с A что-то случится (потеря доступа, увольнение) —
восстанавливать доступ будет некому, потому что и создание нового
пользователя (`POST /users`), и выдача admin (`PATCH /users/{id}`) сами
требуют уже существующей admin-сессии.

**Фикс:** перед `UPDATE`, если запрос снимает admin ИЛИ active с
пользователя, который сейчас является активным админом — считаем
остальных активных админов (`id != $1`); если ноль — 409 "cannot
remove the last remaining admin".

### 4.6. ✅ Разрешения subuser/API-key принимаются без валидации по whitelist
`SubuserHandler.Create/Update` и `APIKeyHandler.Create` берут
`permissions []string` от клиента и сохраняют как есть (просто
`json.Marshal` в jsonb-колонку) — без проверки, что каждая строка
входит в реальный список пермишенов (`auth.Perm*`/`SUBUSER_PERMISSIONS`
на фронте). Само по себе не даёт лишних прав (все проверки идут через
`HasKeyPermission`/`CanAccessServer`, которые матчат по конкретной
строке), но позволяет владельцу сервера захламить permissions
случайными опечатками, которые тихо ничего не делают — стоило бы хотя
бы отбрасывать неизвестные строки с ошибкой, чтобы не создавалось
ложное чувство "я выдал право", когда оно не сработает ни разу.

**Фикс:** добавлен `auth.IsValidPermission(code)` (whitelist из тех же
`Perm*` констант) и вызывается в `SubuserHandler.Create/Update` и
`APIKeyHandler.Create` — неизвестная строка теперь 400, а не тихо
сохраняется. Сверил с фронтовым `SUBUSER_PERMISSIONS` — списки
совпадают 1:1, ничего легитимное не сломается.

### 4.7. ✅ `VersionHandler.CheckUpdate` не проверяет формат ответа перед сравнением
`latestReleasedVersion` читает тело `raw.githubusercontent.com/.../VERSION`
и сравнивает как строку без проверки, что это вообще похоже на semver.
Если CDN/GitHub временно отдаст HTML страницу ошибки со статусом < 400
(например, кастомная страница на edge-узле) — `updateAvailable` станет
true на любую дичь, и UI покажет "доступно обновление" с мусорной
версией. Не критично, но стоит хотя бы regexp-проверить формат перед
сравнением.

**Фикс:** `latestReleasedVersion` теперь гоняет ответ через
`regexp` `^\d+\.\d+\.\d+(-[0-9A-Za-z.-]+)?$` и возвращает ошибку, если
не похоже на semver — раньше до сравнения дело не доходит.

### 4.8. ✅ `GET /nodes/{id}/status` — ещё одна дыра в приватности нод
`router.go:109` — `r.Get("/nodes/{id}/status", nodeHandler.Status)` без
`RequireAdmin` и без проверки `is_public`. `NodeHandler.Status` просто
пингует демон по любому переданному `id` и отдаёт `online`/`error` —
любой залогиненный пользователь может перебрать id нод и узнать, какие
приватные ноды сейчас живы, плюс получить сырую строку ошибки
подключения (может содержать внутренние детали сети/FQDN демона).
Тот же класс бага, что 4.1/4.2 — забыли гейт при добавлении фичи
приватности нод.

**Фикс:** в отличие от 4.1/4.2, этот эндпоинт используется ТОЛЬКО на
админской странице Nodes (`Nodes.tsx`, обычные юзеры её не видят) —
здесь можно было просто добавить `auth.RequireAdmin`, что и сделано.

### 4.9. ✅ `DELETE /servers/{uuid}?force=true` доступен владельцу и может тихо испортить учёт капасити ноды
`ServerHandler.Delete` разрешён владельцу сервера (не только админу).
С `?force=true`, если демон недоступен или отказал в удалении —
запись всё равно удаляется из `servers`. Но реальный контейнер на ноде
при этом мог остаться запущенным (демон ведь не подтвердил удаление).
После этого `SELECT SUM(memory_mb), SUM(disk_mb) FROM servers WHERE
node_id = ...` в `ServerHandler.Create` уже не видит эту память/диск —
они физически заняты на ноде, но панель считает их свободными. Другой
пользователь может тогда создать новый сервер, который в сумме
реально переполнит ноду (OOM/диск), хотя по учёту в БД всё в пределах
capacity. Стоило бы либо сделать `force=true` админ-only, либо не
списывать сервер из капасити, если демон не подтвердил удаление.

**Фикс:** `force := claims.IsAdmin && r.URL.Query().Get("force") ==
"true"` — обычный владелец теперь просто не может активировать
force-режим (получит обычную ошибку "node unavailable" вместо тихого
удаления записи), только админ.

### 4.10. ⏳ ОТЛОЖЕНО — WS-токен передаётся в query string URL
`router.go: authenticateWS` берёт JWT access-токен из
`r.URL.Query().Get("token")` для `/ws/servers/{uuid}` и
`/ws/servers/{uuid}/console`. Токен в URL — известная плохая практика:
он оседает в логах доступа nginx/прокси, в истории браузера, потенциально
в Referer, если со страницы с открытым сокетом когда-либо произойдёт
переход по внешней ссылке. Плюс `websocket.Upgrader{CheckOrigin: func(r)
bool { return true }}` в `backend/internal/ws/hub.go` отключает
проверку Origin как второй рубеж защиты. Токен всё ещё нужен знать
атакующему (не куки, не подхватывается автоматически браузером), так что
это не CSWSH напрямую, но сама утечка токена через URL — отдельный
жизнеспособный риск (например, через access-логи на нодах с nginx,
которые уже стоят для доменной фичи). Стоило бы передавать токен через
`Sec-WebSocket-Protocol` заголовок или через отдельный short-lived
WS-ticket вместо самого access-токена.

**Почему не пофикшено в этот заход:** единственный настоящий фикс —
подпротокол/WS-тикет — требует одновременных изменений и на бэке
(`authenticateWS`, `websocket.Upgrader.Subprotocols`), и на фронте
(`connectServerSocketWithRetry`/`connectConsoleSocketWithRetry`), с
риском тихо сломать реконнект-логику, которую чинили отдельно в этой
же сессии. Раз просили "быстро" — оставляю как реальный, но
единственный ещё открытый пункт, чтобы не гнать рискованный WS-рефактор
без отдельной проверки в браузере.

### 4.11. ✅ Смена пароля / отключение 2FA / деактивация юзера НЕ отзывают уже выданные токены
JWT в `backend/internal/auth/jwt.go` полностью stateless — `Parse`
проверяет только подпись и `exp`, никакого token-version/blacklist
поля нет. Access-токен живёт 15 минут (`config.go: AccessTokenTTL`),
это ещё туда-сюда, но refresh-токен живёт 30 дней
(`RefreshTokenTTL`) и НИГДЕ не хранится/не отзывается на сервере.
Значит:
- Если у юзера увели refresh-токен (украденное устройство, XSS,
  забытая сессия) и он "защищается" — меняет пароль
  (`AuthHandler.ChangePassword`) или отключает 2FA
  (`TwoFAHandler.Disable`) — украденный refresh-токен как был
  валиден, так и остаётся валиден ещё до 30 дней, потому что
  `Refresh` проверяет только `is_active` в БД, а не то, что пароль
  сменился после выпуска токена.
- Админ, деактивировавший или разжаловавший другого юзера
  (`UserHandler.Update`), должен ждать до 15 минут, пока не истечёт
  его access-токен, прежде чем права реально перестанут работать —
  никакого немедленного kill-switch нет.

Обычно это чинится либо version/generation полем в `users`
(инкрементится при смене пароля/деактивации, кладётся в claims и
сверяется при каждом запросе), либо таблицей отозванных refresh-
токенов. Сейчас ничего из этого нет.

**Фикс:** добавлена колонка `users.token_version` (миграция
`0008_user_token_version.sql`, default 0) + поле `TokenVersion` в JWT
claims. `Login`/`Refresh` кладут в токен текущий `token_version` из
БД; `Refresh` дополнительно сверяет `claims.TokenVersion` с текущим
значением в БД и отклоняет устаревший refresh-токен (`401 session
revoked`). `token_version` инкрементится в: `ChangePassword`,
`TwoFAHandler.Disable`, `UserHandler.ResetPassword` (админ сбросил
чужой пароль) и `UserHandler.Update` (когда снимают admin/active).
Осознанно НЕ трогал `auth.Middleware` — access-токен по-прежнему
живёт свои 15 минут без похода в БД на каждый запрос (это отдельный,
более дешёвый компромисс, а не баг: цена входа "проверять DB на
каждый запрос" не стоит того ради 15-минутного окна). Главная дыра —
30-дневный refresh-токен, переживающий смену пароля — закрыта
полностью.

### 4.12. ✅ Rate-limit на логин — только по IP, не по аккаунту
`AuthHandler.checkLoginRateLimit` ключуется как
`"ratelimit:login:" + activity.RequestIP(r)` — 10 попыток / 15 минут
на IP. У конкретного аккаунта отдельного лимита нет: атака с
множества IP (ботнет, ротация прокси) может перебирать пароль ОДНОГО
и того же email без какого-либо накопительного ограничения на сам
аккаунт, потому что лимит есть только на источник, а не на цель.
Стоит добавить второй, менее строгий, но per-email лимит
(`ratelimit:login:email:<email>`) в дополнение к per-IP.

**Фикс:** добавлен именно такой второй лимит — 20 попыток/15 минут по
`ratelimit:login:email:<lowercased email>`, в дополнение к
существующему per-IP лимиту (10/15 минут). Оба должны пройти, иначе
429 с соответствующим сообщением.

### 4.13. ✅ Удаление сервера НИКОГДА не чистит его файлы и бэкапы на диске ноды — утечка места навсегда
Самая крупная находка этого захода. `daemon/internal/api/handlers.go:
Delete` (вызывается из `ServerHandler.Delete` на панели) делает только
`h.Docker.RemoveContainer(...)` — а это `ContainerRemove(..., Force:
true, RemoveVolumes: true)`. `RemoveVolumes` тут ничего не даёт,
потому что папка сервера примонтирована как **bind mount**
(`docker/manager.go: CreateContainer` → `mount.Mount{Type:
mount.TypeBind, Source: m.serverVolumePath(...), Target:
"/home/container"}`), а не как named docker volume — bind-mounts
`RemoveVolumes` не трогает вообще. Проверил через `grep` по всему
демону: `os.RemoveAll` для `ServerVolumePath` не вызывается НИГДЕ,
кроме как внутри `files.Delete`/`files.Rename` (это по запросу юзера
на конкретный путь, не на весь каталог сервера).

Итог: при удалении сервера (обычное удаление, `?force=true`, через
UI — не важно) на диске ноды остаётся:
1. Вся папка `dataDir/<server-uuid>/` со всеми файлами игрового
   сервера — навсегда;
2. Весь каталог `backupDir/<server-uuid>/` со всеми `.tar.gz`
   бэкапами этого сервера — тоже навсегда (бэкапы, в отличие от
   удаления одного бэкапа через `ServerBackupHandler.Delete`, при
   удалении самого сервера никак не подчищаются).

На панели, которая теперь активно занимается capacity/overallocation
нод (в этой же сессии), это довольно серьёзно: диск ноды будет
физически заканчиваться от "удалённых" серверов, о существовании
которых панель полностью забыла, и админ не увидит причину в UI —
только зайдя на ноду руками и посмотрев `du -sh` по датадиру. Нужно
добавить `os.RemoveAll(m.ServerVolumePath(serverUUID))` (и то же для
папки бэкапов) в `Handlers.Delete` после успешного
`RemoveContainer`.

**Фикс:** именно это и добавлено в `daemon/internal/api/handlers.go:
Delete` — после успешного `RemoveContainer` идёт
`os.RemoveAll(h.Docker.ServerVolumePath(serverUUID))`, и следом (если
`BackupDir` задан) `os.RemoveAll(filepath.Join(h.BackupDir,
serverUUID.String()))`. Обе ошибки от `RemoveAll` бэкапов игнорируются
намеренно (best-effort, чтобы отсутствие бэкапов не блокировало
удаление), ошибка от `RemoveAll` файлов сервера возвращается вызывающему
(контейнер уже удалён к этому моменту, так что 500 здесь — сигнал
руками почистить диск, а не блокирующая проблема).
